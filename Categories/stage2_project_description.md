# Проект автоматического категорирования аптечных товаров

> Бизнес-описание для заказчика. Техническая реализация, контракты БД и workflow — в [`PROJECT.md`](PROJECT.md), журнал задач — в [`stage2_workflow_plan.md`](stage2_workflow_plan.md).

## Общая идея проекта

Проект направлен на создание системы, которая автоматически распределяет аптечные товары по правильным категориям на основе их названий, описаний и справочника категорий.[cite:1] Система строится как многоэтапный процесс принятия решений: сначала товар быстро оценивается по правилам, затем анализируется основной AI-моделью, после чего сложные случаи проходят дополнительные уточняющие этапы и при необходимости передаются человеку на финальную проверку.[cite:1]

Главная задача проекта — сократить долю ручной работы, повысить единообразие категорирования и сделать весь процесс прозрачным для контроля качества.[cite:1] Вместо одного рискованного автоматического шага используется последовательная схема проверки, в которой каждый товар либо классифицируется автоматически, либо направляется на следующий уровень анализа.[cite:1]

## Как будет работать система

Система работает не по одному товару, а партиями: в обработку запускается группа товаров по заданному количеству, после чего для всей этой группы создаётся отдельный запуск обработки.[cite:1] Уже подтверждено, что система поддерживает batch-обработку и сохраняет результаты по каждому запуску, включая количество успешно обработанных товаров, ошибок и товаров, требующих дополнительного внимания; в тестовом прогоне использовалась партия из 5 товаров.[cite:1]

Для каждого такого запуска сохраняется отдельная запись, чтобы можно было видеть, сколько товаров поступило в работу, каков итог по партии и какие товары потребовали дополнительных шагов.[cite:1] Это позволяет управлять процессом не вручную по одному товару, а на уровне полноценных рабочих сессий обработки.[cite:1]

## Этап 1. Подготовка и предварительный отбор

На первом этапе система приводит данные товара к единому виду: собирает описание, выделяет ключевые признаки и формирует удобное представление для дальнейшего анализа.[cite:1] Одновременно строится предварительный список наиболее вероятных категорий, чтобы сузить область поиска и не сравнивать товар со всем справочником сразу.[cite:1]

На выходе этого этапа каждый товар уже имеет нормализованное описание и первичный shortlist возможных категорий.[cite:1] Это не финальное решение, а подготовка к основному автоматическому категорированию.[cite:1]

## Этап 2. Основное автоматическое решение

После подготовки включается основная AI-модель, которая предлагает категорию, уровень уверенности и короткое объяснение выбора.[cite:1] Если решение выглядит достаточно уверенным и проходит внутреннюю проверку качества, категория сразу фиксируется как финальная.[cite:1]

Если модель отвечает неуверенно, неполно, противоречиво или вообще не может дать пригодный результат, товар не получает сомнительную финальную категорию.[cite:1] Вместо этого система переводит его на следующий этап уточнения или сразу в ручную проверку, в зависимости от характера проблемы.[cite:1]

## Этап 3. Уточнение сложных случаев

Если основной автоматический раунд не дал надёжного результата, запускается дополнительный сценарий из двух шагов.[cite:1] Сначала система определяет более общий уровень принадлежности товара — например, направление, семейство или укрупнённую ветку классификации, а затем внутри этой ветки строит новый shortlist категорий для повторного выбора.[cite:1]

Такой подход особенно важен для неоднозначных товаров, где сразу выбрать точную категорию трудно, но можно достаточно надёжно определить более узкий сегмент поиска.[cite:1] В результате система получает второй шанс принять корректное решение уже в более ограниченном и релевантном наборе категорий.[cite:1]

## Этап 4. Разбор спорных ситуаций

Если после нескольких автоматических шагов остаются сомнения, предусмотрен дополнительный слой проверки спорных кейсов.[cite:1] Он используется в ситуациях, когда разные этапы предлагают разные варианты, когда уверенность слишком низкая или когда товар по-прежнему нельзя безопасно отнести к одной категории.[cite:1]

Задача этого этапа — либо выбрать лучший вариант из нескольких, либо честно признать, что товар нужно передать специалисту.[cite:1] Это снижает риск неверной категоризации в тех случаях, где автоматике недостаточно данных или уверенности.[cite:1]

## Этап 5. Ручная верификация

Для наиболее сложных товаров предусмотрена ручная проверка через Telegram.[cite:1] Специалист получает карточку товара с его описанием, вариантами, предложенными системой, и краткими пояснениями, после чего может подтвердить категорию или выбрать другую.[cite:1]

Это означает, что человек подключается только к действительно спорным случаям, а основной поток товаров проходит автоматически.[cite:1] Такой подход позволяет сохранить качество результата без перегрузки команды ручной обработкой.[cite:1]

## Порядок принятия решений

Система принимает решения по простому и понятному принципу: если решение надёжное, оно фиксируется автоматически; если есть сомнения, товар отправляется на следующий уровень проверки.[cite:1] В результате логика выглядит так: подготовка товара -> основное автоматическое решение -> уточняющий разбор -> разбор спорных случаев -> ручная верификация при необходимости.[cite:1]

Для бизнеса это важно потому, что система не пытается “додумать” ответ там, где в нём нет уверенности.[cite:1] Вместо этого она управляемо снижает риск ошибок и направляет спорные товары в отдельный, контролируемый процесс проверки.[cite:1]

## Что реализовано уже сейчас

На текущем этапе уже собран рабочий каркас системы, который покрывает основной автоматический раунд и всю инфраструктуру учёта результатов.[cite:1] Уже реализованы запуск обработки партиями, передача общего контекста через весь workflow, нормализация товара, формирование предварительного shortlist, вызов основной AI-модели, проверка её ответа и запись результата в базу данных.[cite:1]

Также уже работает подробное логирование каждого шага, чтобы по каждому товару было видно, как именно система принимала решение и что произошло на каждом этапе.[cite:1] Кроме того, реализовано завершение пакетного запуска с фиксацией итогов по партии, включая статистику по количеству успешно обработанных товаров и товаров, требующих дополнительного внимания.[cite:1]

Отдельно важно, что архитектура уже подготовлена под следующие этапы: в базе предусмотрены поля для результатов уточняющих шагов и повторного анализа, а система логирования уже совместима с будущими стадиями дополнительной проверки и ручной верификации.[cite:1] Это означает, что текущая реализация — не отдельный прототип, а фундамент для полной рабочей схемы категорирования.[cite:1]

## Какие результаты ожидаются на каждом этапе

| Этап | Что делает система | Какой результат получает заказчик |
|---|---|---|
| Подготовка товара | Приводит описание к единому виду и собирает признаки товара [cite:1] | Товар готов к дальнейшему анализу, снижен уровень «шума» в данных [cite:1] |
| Предварительный shortlist | Формирует список наиболее вероятных категорий [cite:1] | Поиск становится быстрее и точнее [cite:1] |
| Основное категорирование | AI-модель выбирает категорию и оценивает уверенность [cite:1] | Простые товары закрываются автоматически [cite:1] |
| Уточняющий разбор | Определяет более общий сегмент и повторно уточняет категорию [cite:1] | Сложные товары получают второй, более точный раунд анализа [cite:1] |
| Проверка спорных случаев | Сравнивает конфликтные или слабые решения [cite:1] | Снижается риск финальной ошибки [cite:1] |
| Ручная верификация | Передаёт спорные товары специалисту [cite:1] | Команда проверяет только действительно сложные случаи [cite:1] |
| Итог по запуску | Подводит результат по всей партии товаров [cite:1] | Видна полная картина: сколько обработано автоматически, сколько ушло на разбор [cite:1] |

## Какой итоговый результат получит заказчик

На выходе заказчик получает не просто модель, которая предлагает категории, а полноценную систему управления категорированием товарного потока.[cite:1] Она позволяет обрабатывать товары партиями, автоматически закрывать значительную часть потока, отдельно маршрутизировать спорные случаи и сохранять полную историю принятия решений.[cite:1]

Практический результат проекта — это повышение скорости обработки товаров, снижение объёма ручной работы и улучшение качества категорирования за счёт многоэтапной проверки.[cite:1] Дополнительно заказчик получает прозрачность: по каждому товару и по каждому запуску можно понять, как именно было принято решение и на каком этапе оно было подтверждено.[cite:1]

## Следующий этап развития

Следующий шаг проекта — довести до полноценной работы уточняющий сценарий для сложных случаев, затем подключить дополнительный проверочный слой и завершить контур ручной верификации через Telegram.[cite:1] После этого система будет закрывать полный цикл: от загрузки партии товаров до сохранения финальной подтверждённой категории с понятной историей принятия решения.[cite:1]
